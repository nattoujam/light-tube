# CLENE Refactoring Journal - lightweight-video-player-python

## 重要な学び

### 1. SQLite接続処理のボイラープレート排除
- **状況**: `VideoStorage` クラスの各メソッドで `sqlite3.connect` と `row_factory` の設定が繰り返されていた。
- **改善**: `contextlib.contextmanager` を使用して `_connection` メソッドを導入。これにより、接続、トランザクション管理（commit/rollback）、およびクローズを一元化できた。
- **正解だった理由**: コード量が削減されるだけでなく、将来的に他のデータベースへの移行や接続オプションの変更が一箇所で済むようになり、保守性が大幅に向上した。

### 2. 複雑な条件分岐の抽出による可読性向上
- **状況**: `select_next_video` メソッドが、複数の再生ルールと動的なSQL構築を含む巨大なメソッドになっていた。
- **改善**: 各再生ルール（関連動画、新着動画等）を `_find_related_unviewed` などのプライベートメソッドに分割した。
- **正解だった理由**: 各ルールのロジックが独立したことで、コードを「読む」際の認知負荷が劇的に下がった。また、特定のルールの調整や新規ルールの追加が容易になった。

### 3. あえて「具体的なまま」にする判断
- **状況**: `id NOT IN (...)` という除外条件の構築など、SQLの一部に重複が見られた。
- **改善**: 当初はヘルパー関数による共通化を検討したが、最終的に各検索メソッド内にSQLを直接記述（インライン化）する構成を採用した。
- **正解だった理由**: ユーザーからのフィードバックに基づき、SQL文を断片化せず一箇所にまとめることで、クエリの意図が即座に理解できるようになり、見通しが向上した。抽象化が常に正義ではなく、可読性のためにあえて具体的なままに留めるべきケースがあることを再確認した。

### 4. AppStateによるドメインモデルのキャッシュとUIの疎結合化
- **状況**: `Tui.render` が毎秒20回（50ms間隔）実行される際、その都度 `select_next_video` によるDBクエリが走っており、パフォーマンス上の懸念があった。また、`main.py` の各所で視聴済みマークのDB更新ロジックが重複していた。
- **改善**:
    1. `AppState` に `next_video` や `selected_video` といった `Video` ドメインモデルを直接保持するキャッシュを導入。
    2. `refresh_app_state` と `mark_video_as_viewed` ヘルパー関数により、状態更新とDB保存のロジックを共通化。
    3. `Tui` クラスから `VideoStorage` への依存を完全に排除し、描画に必要なデータはすべて `AppState` から取得するように変更。
- **正解だった理由**: 描画ループ内の冗長なDBクエリを完全に排除できた。また、UIレイヤーがストレージ（SQLite）の詳細を知る必要がなくなったため、将来のGoへの移行やストレージの変更がより容易になった。

### 5. UI状態のAppStateへの集約によるロジックの分離
- **状況**: `main.py` の `handle_input` 関数内で、キー入力の判定、UIの内部状態（選択インデックスやヘルプ表示フラグ）の更新、およびアプリケーションの状態遷移が混在していた。
- **改善**: `AppState` に `selected_idx`, `show_help`, `busy_until` を追加し、カーソル移動やヘルプ表示の切り替えを `Event` を介した状態遷移として実装した。`Tui` クラスは `AppState` の情報のみを見て描画するように変更した。
- **正解だった理由**: 入力処理（Controller）、状態管理（Model/State）、描画処理（View）の責務が明確に分離された。これにより、将来的に TUI から GUI へ変更する場合でも UI 層の差し替えだけで対応が可能になり、またインタラクションのロジックをユニットテストで検証できるようになった。

### 6. メインロジックのクラス化による凝集度の向上
- **状況**: `app/main.py` が多数のグローバル関数で構成されており、`AppState` や `VideoStorage` などの依存オブジェクトを毎回引数で渡す必要があった。
- **改善**: `VideoPlayerApp` クラスを導入し、アプリケーションのライフサイクルと主要コンポーネントをカプセル化した。
- **正解だった理由**: 状態更新や入力処理がメソッドとして整理され、`self` を介して各コンポーネントにアクセスできるようになったため、コードの見通しが劇的に改善した。

### 7. 不要な抽象化レイヤーの削除
- **状況**: `app/next_logic.py` が `VideoStorage` の単なるラッパーとして存在していた。
- **改善**: ラッパーを廃止し、ストレージのメソッドを直接呼び出すように変更した。
- **正解だった理由**: 意味のない仲介役を排除することで、関心の分離を維持しつつコールスタックを単純化できた。

### 8. オーケストレーション層における巨大メソッドの分解と責務の明確化
- **状況**: `VideoPlayerApp` クラスの `handle_input` や `update_background_status` が多岐にわたる責務を持つ巨大なメソッドになっていた。
- **改善**: プレイヤーイベント処理、バックグラウンド同期処理、ビデオ取得・保存、複雑な対話フローをそれぞれ独立したメソッドに分離。
- **正解だった理由**: 「何をしているか」がメソッド名で明確になり、メインループの制御フローが劇的に読みやすくなった。

### 9. TUI描画ロジックの階層化と責務分割
- **状況**: `Tui` クラスの描画メソッドが肥大化し、座標計算とロジックが混在していた。
- **改善**: スクロールオフセット管理、行単位の描画、およびステータスメッセージ生成を独立したメソッドとして抽出。
- **正解だった理由**: 描画の「宣言的な流れ」がメインメソッドで明確になり、個別の詳細に惑わされずに UI 構造を把握できるようになった。

### 10. ドメインモデルによる一貫性の強制
- **状況**: チャンネル登録フローにおいて、`namedtuple` による一時的な「ダミーオブジェクト」が使用されていた。
- **改善**: `app/models.py` で定義された `Channel` クラスを直接インスタンス化して使用するように変更。
- **正解だった理由**: アプリケーション全体で同じデータ構造を扱うことで、型安全性と将来の Go への移行時の整合性を確保できた。

### 11. ディスパッチャパターンによる入力処理の整理
- **状況**: `VideoPlayerApp.handle_input` メソッドが巨大な `if-elif` チェーンで構成されていた。
- **改善**: キーコードとメソッドを紐付ける `action_map`（辞書）を導入し、ディスパッチャ形式で呼び出すように変更。
- **正解だった理由**: 入力検知とアクション実行が明確に分離され、将来的な機能拡張に対する耐性が高まった。

### 12. 公開メソッドによるカプセル化の徹底
- **状況**: `Repository` クラスから `VideoStorage` クラスのプライベートな接続管理メソッドを直接呼び出していた。
- **改善**: `VideoStorage` に適切な公開メソッドを追加し、具体的なデータ取得ロジックをストレージ層へ集約。
- **正解だった理由**: 各レイヤの責務が明確になり、上位レイヤが下位レイヤの内部実装を知る必要がなくなった（デメテルの法則の遵守）。

### 13. SQLパラメータバインディングの名前付きパラメータ採用
- **状況**: SQL パラメータ（位置指定 `?`）の手動展開により、記述が重複しバグが発生しやすい状態だった。
- **改善**: `Video` オブジェクトを辞書に変換する `to_dict` を活用し、SQL を名前付きパラメータ（`:id` 等）形式に変更。
- **正解だった理由**: SQL 文とデータの対応が明示的になり、保守性が向上した。

### 14. 目的に応じたクエリ実行ヘルパーの導入とボイラープレート排除
- **状況**: `VideoStorage` 内の各取得メソッドで、接続管理と `fetchone/fetchall` の記述が繰り返されていた。また、単一取得時に `fetchall` を使用する非効率な設計も見られた。
- **改善**: 接続管理を隠蔽し、用途別に最適化された3種類のヘルパー（`_fetch_all`, `_fetch_one`, `_run`）を導入。
- **正解だった理由**: 無駄なDBアクセス（単一取得時の全行フェッチ）を排除し、パフォーマンスを最適化できた。また、用途がメソッド名で明示されるため、コードの意図がより明確になり、ボイラープレートも最小限に抑えられた。
- **学んだ点**: `_fetch_one` で `MAX()` 等の集計結果を取得する際は、SQL 側で `AS alias` を指定し、Python 側でキー名で参照することで、インデックス指定 (`row[0]`) によるマジックナンバーを排除でき、可読性がさらに向上する。

### 15. JOINによるドメインモデル間の強力なリレーションシップの実現
- **状況**: `Video` モデルが `channel_id` という単なる ID しか持っておらず、リレーションが希薄だった。
- **改善**: `Video.channel` の型を `Channel` オブジェクトに変更し、ストレージ層の全ての動画取得クエリに `JOIN channels` を導入。
- **正解だった理由**: ドメインモデルとしての整合性が極限まで高まり、UI 層での無駄な逆引き処理を一掃できた。

### 16. AppStateへのアイテムリスト管理集約によるUIロジックの簡素化
- **状況**: `Tui` や `main.py` で、現在のタブが「ビデオ」か「チャンネル」かを判定し、それぞれのリストに対してインデックスアクセスするロジックが重複していた。
- **改善**: `AppState` に `current_items` および型安全な `highlighted_video` / `highlighted_channel` プロパティを導入。
- **正解だった理由**: UI層やコントローラー層から煩雑なインデックス管理や境界チェックを一掃できた。また、`current_items` を介して描画ループ内のロジックを共通の `_draw_line` ヘルパーに集約しつつ、利用側（VideoPlayerApp等）では型情報を維持したままアイテムを取得できるため、可読性と堅牢性の両立が達成された。
