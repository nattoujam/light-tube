# CLENE Refactoring Journal - lightweight-video-player-python

## 重要な学び

### 1. SQLite接続処理のボイラープレート排除
- **状況**: `VideoStorage` クラスの各メソッドで `sqlite3.connect` と `row_factory` の設定が繰り返されていた。
- **改善**: `contextlib.contextmanager` を使用して `_connection` メソッドを導入。これにより、接続、トランザクション管理（commit/rollback）、およびクローズを一元化できた。
- **正解だった理由**: コード量が削減されるだけでなく、将来的に他のデータベースへの移行や接続オプションの変更が一箇所で済むようになり、保守性が大幅に向上した。

### 2. 複雑な条件分岐の抽出による可読性向上
- **状況**: `select_next_video` メソッドが、複数の再生ルールと動的なSQL構築を含む巨大なメソッドになっていた。
- **改善**: 各再生ルール（関連動画、新着動画等）を `_find_related_unviewed` などのプライベートメソッドに分割した。
- **正解だった理由**: 各ルールのロジックが独立したことで、コードを「読む」際の認知負荷が劇的に下がった。また、特定のルールの調整や新規ルールの追加が容易になった。

### 3. あえて「具体的なまま」にする判断
- **状況**: `id NOT IN (...)` という除外条件の構築など、SQLの一部に重複が見られた。
- **改善**: 当初はヘルパー関数による共通化を検討したが、最終的に各検索メソッド内にSQLを直接記述（インライン化）する構成を採用した。
- **正解だった理由**: ユーザーからのフィードバックに基づき、SQL文を断片化せず一箇所にまとめることで、クエリの意図が即座に理解できるようになり、見通しが向上した。抽象化が常に正義ではなく、可読性のためにあえて具体的なままに留めるべきケースがあることを再確認した。

### 4. AppStateによるドメインモデルのキャッシュとUIの疎結合化
- **状況**: `Tui.render` が毎秒20回（50ms間隔）実行される際、その都度 `select_next_video` によるDBクエリが走っており、パフォーマンス上の懸念があった。また、`main.py` の各所で視聴済みマークのDB更新ロジックが重複していた。
- **改善**:
    1. `AppState` に `next_video` や `selected_video` といった `Video` ドメインモデルを直接保持するキャッシュを導入。
    2. `refresh_app_state` と `mark_video_as_viewed` ヘルパー関数により、状態更新とDB保存のロジックを共通化。
    3. `Tui` クラスから `VideoStorage` への依存を完全に排除し、描画に必要なデータはすべて `AppState` から取得するように変更。
- **正解だった理由**: 描画ループ内の冗長なDBクエリを完全に排除できた。また、UIレイヤーがストレージ（SQLite）の詳細を知る必要がなくなったため、将来のGoへの移行やストレージの変更がより容易になった。
- **学んだ点**: キャッシュを導入する場合、状態遷移（再生開始、終了、タブ切り替えなど）の各タイミングで確実に `refresh_app_state` を呼び出す必要があり、特に `LAUNCHING` 状態（再生準備中）のような過渡的な状態での「現在の動画」の扱いに注意が必要であった。

### 5. UI状態のAppStateへの集約によるロジックの分離
- **状況**: `main.py` の `handle_input` 関数内で、キー入力の判定、UIの内部状態（選択インデックスやヘルプ表示フラグ）の更新、およびアプリケーションの状態遷移が混在していた。
- **改善**: `AppState` に `selected_idx`, `show_help`, `busy_until` を追加し、カーソル移動やヘルプ表示の切り替えを `Event` を介した状態遷移として実装した。`Tui` クラスは `AppState` の情報のみを見て描画するように変更した。
- **正解だった理由**: 入力処理（Controller）、状態管理（Model/State）、描画処理（View）の責務が明確に分離された。これにより、将来的にTUIからGUIへ変更する場合でもUI層の差し替えだけで対応が可能になり、またインタラクションのロジック（インデックスの範囲チェックなど）をユニットテストで検証できるようになった。

### 6. メインロジックのクラス化による凝集度の向上
- **状況**: `app/main.py` が多数のグローバル関数で構成されており、`AppState` や `VideoStorage` などの依存オブジェクトを毎回引数で渡す必要があった。
- **改善**: `VideoPlayerApp` クラスを導入し、アプリケーションのライフサイクルと主要コンポーネントをカプセル化した。
- **正解だった理由**: 状態更新や入力処理がメソッドとして整理され、`self` を介して各コンポーネントにアクセスできるようになったため、コードの見通しが劇的に改善した。また、`_mark_and_transition` のようなプライベートメソッドを抽出することで、視聴済みマークの付与と状態遷移という定型パターンの重複を排除（DRY）できた。

### 7. 不要な抽象化レイヤーの削除
- **状況**: `app/next_logic.py` が `VideoStorage` の単なるラッパーとして存在していた。
- **改善**: ラッパーを廃止し、ストレージのメソッドを直接呼び出すように変更した。
- **正解だった理由**: 意味のない仲介役を排除することで、関心の分離を維持しつつコールスタックを単純化できた。Goへの移行を見据えた際にも、より直接的なデータアクセス構造の方が設計が明快になる。

### 8. オーケストレーション層における巨大メソッドの分解と責務の明確化
- **状況**: `VideoPlayerApp` クラスの `handle_input` や `update_background_status` が、キー入力のディスパッチ、状態遷移、外部API呼び出し（ビデオ取得）、およびプレイヤー監視など、多岐にわたる責務を持つ巨大なメソッドになっていた。
- **改善**:
    1. プレイヤーイベント処理 (`_process_mpv_events`) とバックグラウンド同期処理 (`_process_background_updates`) を独立したメソッドに分離。
    2. ビデオ取得・保存の定型処理を `_sync_channel_videos` として共通化。
    3. 複雑な対話フロー（チャンネル登録）を `_run_registration_flow` にカプセル化。
- **正解だった理由**: 「何をしているか」がメソッド名で明確になり、メインループの制御フローが劇的に読みやすくなった。また、`_sync_channel_videos` の導入により、新規取得、履歴取得、登録時の初期取得という3つの文脈で重複していたロジックをDRYに保つことができた。
- **学んだ点**: TUIのような対話型アプリケーションでは、単一の入力（キー押下）が複雑な副次作用（外部通信や複数の画面更新）を伴うことが多く、これらを早期にプライベートメソッドへ切り出すことが、スパゲッティコード化を防ぐ鍵となる。

### 9. TUI描画ロジックの階層化と責務分割
- **状況**: `Tui` クラスの `draw_main_area` や `draw_footer` が、座標計算、文字列フォーマット、条件付き属性適用、および実際の描画処理をすべて含んでおり、肥大化していた。
- **改善**:
    1. スクロールオフセット管理 (`_adjust_scroll`)、行単位の描画 (`_draw_video_line`)、およびステータスメッセージ生成 (`_get_status_text`) を独立したメソッドとして抽出。
- **正解だった理由**: 描画の「宣言的な流れ」がメインメソッドで明確になり、個別の詳細（タイトルの切り詰めルールやスクロールの境界判定など）に惑わされずに UI 構造を把握できるようになった。

### 10. ドメインモデルによる一貫性の強制
- **状況**: チャンネル登録フローにおいて、`namedtuple` による一時的な「ダミーオブジェクト」が使用されており、ドメインモデル (`Channel`) との不一致や型安全性の欠如が懸念された。
- **改善**: `app/models.py` で定義された `Channel` クラスを直接インスタンス化して使用するように変更。
- **正解だった理由**: アプリケーション全体で同じデータ構造を扱うことで、将来の Go への移行時にも構造体定義の対応が 1:1 で明確になる。また、インラインでの構造定義を排除することで、コードの凝集度が高まった。

### 11. ディスパッチャパターンによる入力処理の整理
- **状況**: `VideoPlayerApp.handle_input` メソッドが巨大な `if-elif` チェーンで構成されており、新しい操作の追加や各操作のロジックの見通しが悪くなっていた。
- **改善**:
    1. 各キー入力に対応する処理を `_on_key_play` などの専用プライベートメソッドに抽出。
    2. キーコードとメソッドを紐付ける `action_map`（辞書）を導入し、ディスパッチャ形式で呼び出すように変更。
- **正解だった理由**: 入力検知（何が押されたか）とアクション実行（何をするか）が明確に分離された。これにより、同じアクションに複数のキーを割り当てることが容易になり、また `handle_input` 自体の見通しが劇的に向上した。Goへの移植を考える際にも、この種のテーブル駆動方式は実装がクリーンになりやすい。
- **学んだ点**: TUIアプリケーションにおいて、キー入力処理は肥大化しやすい「複雑性の温床」である。早期にディスパッチャパターンを適用することで、将来的な機能拡張に対する耐性が高まる。
